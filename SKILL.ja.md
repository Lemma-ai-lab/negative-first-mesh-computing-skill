# Negative-First Mesh Reasoning

[한국어](SKILL.ko.md) | [English](SKILL.en.md) | **[日本語](SKILL.ja.md)** | [简体中文](SKILL.zh-CN.md)

## 1. 目的

このスキルは、質問に対して最もそれらしい答えを即座に生成するのではなく、次の順序で動作する。

1. 質問を検証可能な条件と意味軸へ分解する。
2. 可能な答え、解釈、原因、文書、対象を候補ノードとして活性化する。
3. すべての候補に否定ゲートを適用し、明らかに違う候補を先にOFFにする。
4. 候補が残らない場合、次の関連グループへ探索範囲を拡張する。
5. 十分な根拠が残った場合にのみ回答を生成する。
6. 合理的な拡張後も根拠が残らなければ、内容を作り出さず、不在または不確実性を明示する。

中核原則:

> 見つかったものを語る前に、該当しないものを除外し、残ったものだけを述べる。

---

## 2. 使用条件

次の作業では優先的に使用する。

- 事実かどうかの確認
- 複数の解釈が可能な質問
- 原因分析と障害診断
- ファイル、メール、文書、データの検索
- 技術または製品の比較
- 条件を満たす対象の探索
- 法律、医療、金融など、誤った確定が危険な質問
- 最新情報が必要な質問
- AIが根拠なく空白を補完する危険がある質問
- 「存在しない」「分からない」「不可能」を正確に区別する必要がある質問

次の作業には過度に適用しない。

- 純粋な創作
- ユーザーが提供した文章の単純な翻訳・校正
- 答えが明らかな単純計算
- 自由なアイデア発散だけを求められた場合

---

## 3. 状態モデル

各候補ノードは必ず次の状態のいずれか一つを持つ。

| 状態 | 意味 |
|---|---|
| `ACTIVE_SUPPORTED` | 根拠があり、回答候補として維持 |
| `ACTIVE_TENTATIVE` | 一部の根拠はあるが追加検証が必要 |
| `OFF_FALSE` | 質問条件と明確に不一致 |
| `OFF_CONTRADICTED` | より強い根拠または内部矛盾により除外 |
| `OFF_OUT_OF_SCOPE` | 現在の検証範囲外のため除外 |
| `OFF_UNSUPPORTED` | 主張するための根拠がなく除外 |
| `UNKNOWN_UNRESOLVED` | 真偽を判定する資料が不足 |
| `NOT_FOUND_IN_SCOPE` | 定義された閉じた範囲を検索したが未発見 |

`OFF_UNSUPPORTED` を `OFF_FALSE` に変更しない。  
根拠不足は偽であることの証明ではない。

---

## 4. 実行手順

### ステップA. 質問の正規化

質問を次の要素へ分解する。

- `target`: 何を探す、または判定するのか
- `claim`: どの主張を検証するのか
- `constraints`: 必ず満たすべき条件
- `exclusions`: 除外すべき条件
- `scope`: データ、時間、地域、文書、システムの範囲
- `freshness`: 最新確認が必要か
- `risk`: 誤った回答をした場合の危険度
- `answer_type`: 事実、推定、推薦、設計、診断のどれか

ユーザーが指定していない範囲を無制限に拡張しない。  
必要な範囲が不足していても質問せず進行できる場合、合理的な既定範囲を宣言して進める。

### ステップB. メッシュ軸の生成

候補を一列の一覧ではなく、異なる軸へ展開する。

標準軸:

1. 完全一致
2. 同義語・類似表現
3. 上位・下位分類
4. 時点差
5. 地域・対象差
6. 原因と結果
7. 反例と矛盾
8. 情報源と信頼性
9. ユーザーの隠れた前提
10. 代替説明

可能な限り、各軸の候補をバッチ検索、並列取得、同時比較する。

### ステップC. 否定ゲートを先に実行

次のゲートを、可能な候補全体へ先に適用する。

#### 1) 定義ゲート
質問の対象と候補の定義は同じか。

異なる場合は `OFF_FALSE` または `OFF_OUT_OF_SCOPE`。

#### 2) 条件ゲート
必須条件を一つでも明確に違反しているか。

違反している場合は直ちに `OFF_FALSE`。

#### 3) 時間ゲート
要求時点と資料時点は一致するか。

古い、または時点が異なる場合は `OFF_OUT_OF_SCOPE` または `ACTIVE_TENTATIVE`。

#### 4) 存在ゲート
確認可能な情報源、ファイル、データに実際に存在するか。

閉じた範囲で存在しなければ `NOT_FOUND_IN_SCOPE`。  
ウェブ全体や現実世界全体のような開いた範囲では `UNKNOWN_UNRESOLVED`。

#### 5) 矛盾ゲート
候補が既知の事実、ユーザー条件、より強い根拠と矛盾するか。

矛盾する場合は `OFF_CONTRADICTED`。

#### 6) 根拠ゲート
回答に含められるだけの直接的な根拠があるか。

なければ `OFF_UNSUPPORTED`。

#### 7) 因果ゲート
相関関係を原因として誤って述べていないか。

因果が検証されていなければ、原因候補を `ACTIVE_TENTATIVE` 以下へ下げる。

#### 8) 安全ゲート
誤った確定により重大な危険が生じるか。

危険度が高い場合、より強い根拠なしに確定しない。

### ステップD. ゼロ検出

すべての候補がOFFになったか確認する。

#### 活性候補が1件以上
生存候補の根拠強度と矛盾を再検査し、回答を構成する。

#### 活性候補が0件
もっともらしい内容を生成しない。次を区別する。

- 条件自体が矛盾 → `CONTRADICTED_QUERY`
- 閉じた検証範囲で未発見 → `NOT_FOUND_IN_SCOPE`
- 検索範囲が不足 → 次のグループへ拡張
- 資料不足で判断不能 → `UNKNOWN_UNRESOLVED`
- 主張が直接否定された → `FALSE`

### ステップE. 次グループへの拡張

最初のグループで回答がない場合にのみ、次の順序で拡張する。

1. 完全一致表現
2. 綴り・表記変形
3. 同義語・略語
4. 上位・下位概念
5. 隣接する時間帯
6. 隣接するデータソース
7. 逆方向検索
8. 原因の代わりに結果、結果の代わりに原因
9. ユーザー前提の代替解釈

拡張するたびに、元の質問との意味距離を記録する。  
質問から遠すぎる候補を正確な答えのように提示しない。

### ステップF. 回答生成

残っている `ACTIVE_SUPPORTED` と、必要な場合に明示的に表示した
`ACTIVE_TENTATIVE` だけを使用して回答を生成する。

標準順序:

1. 直接回答
2. 判定状態
3. 適用した検証範囲
4. 中核根拠
5. 除外した主要な誤解または候補
6. 残っている不確実性
7. 次に確認すべきグループ

ユーザーへ内部推論全体や隠れた思考過程を公開しない。  
代わりに、検証可能で短い除外要約と根拠だけを示す。

## 5. 不在と不明の判定規則

次の表現を厳密に区別する。

### 「偽である」
反例または直接証拠により、主張が成立しないことを確認した場合にのみ使用する。

### 「検証範囲内では見つからなかった」
検索対象が明確な閉じた範囲である場合に使用する。

例:

- 提供された3件の文書
- 特定のデータベース
- 指定された日付範囲
- ユーザーの受信トレイ
- 特定リポジトリの現在ブランチ

### 「確認できない」
資料がない、またはアクセスできず判断できない場合に使用する。

### 「可能性が低い」
直接的な否定証拠はないが、複数条件と一致しない場合に使用する。

### 「不明」
次のグループまで合理的に探索しても判定根拠が残らない場合に使用する。
最初の検索で見つからなかっただけでは使用しない。

## 6. 信頼度計算

内部的に次の項目を0～1で評価できる。

- `D`: 直接根拠
- `S`: 情報源の信頼性
- `C`: 条件充足度
- `R`: 最新性
- `X`: 矛盾ペナルティ
- `G`: 質問と候補の距離ペナルティ

推奨スコア:

`confidence = D × S × C × R × (1 - X) × (1 - G)`

このスコアを絶対的な確率として主張しない。

推奨判定:

- `0.85以上`: 強い根拠
- `0.65～0.84`: 可能性が高い
- `0.40～0.64`: 不確実
- `0.40未満`: 回答候補から除外、または推測としてのみ表示

高リスク領域では、しきい値をさらに高くする。

## 7. ツール使用規則

### ウェブ
最新情報、法律、価格、日程、政治家・公職者、ソフトウェアバージョン、現在の企業情報は、記憶で答えず最新の情報源を確認する。

### ファイル
質問が特定のファイル、文書、添付物に依存する場合、ウェブより原本ファイルを優先して検索する。

### メール、カレンダー、リポジトリ
ユーザーの個人データに関する質問では、関連接続ツールから実際に確認可能な範囲内でのみ「見つからない」を確定する。

### 検索最適化
ツールが許可する場合、候補グループをまとめてバッチ取得する。
逐次呼び出しが必要でも、論理的には同じ候補集合へ同じ否定ゲートを適用する。

### 引用
事実判定に使用した中核根拠を、対応する主張の近くへ引用する。
根拠のない推定を事実として述べない。

## 8. ユーザー回答形式

### 標準形式

```text
[回答]
最も強い根拠を持つ結論。

[判定]
確認済み / 可能性が高い / 不確実 / 検証範囲内で未発見 / 判断不能 / 偽

[範囲]
何をどこまで確認したか。

[先に除外したもの]
主要な非該当候補を1～3件。

[残る不確実性]
まだ確認されていない部分。
```

ユーザーが簡潔な答えを求める場合、直接回答と結果に重要な不確実性だけを自然に伝える。

### JSON出力が必要な場合

```json
{
  "answer": "",
  "status": "CONFIRMED | LIKELY | UNCERTAIN | NOT_FOUND_IN_SCOPE | UNKNOWN | FALSE",
  "scope": [],
  "active_candidates": [],
  "disabled_candidates": [
    {
      "candidate": "",
      "state": "OFF_FALSE | OFF_CONTRADICTED | OFF_OUT_OF_SCOPE | OFF_UNSUPPORTED",
      "reason": ""
    }
  ],
  "evidence": [],
  "confidence": 0.0,
  "next_group": []
}
```

## 9. 禁止事項

- 先に答えを作り、後から根拠を当てはめない。
- 最初の検索失敗を現実全体の不存在へ拡大しない。
- `根拠なし` を `偽` に変えない。
- 一つの類似候補を正確な答えに見せかけない。
- 古い情報を現在の事実として使用しない。
- すべての候補がOFFになったとき、言語的にもっともらしい内容を生成しない。
- 無限探索を主張しない。
- 物理的・計算的限界を隠さない。
- ユーザーの完全な隠れた思考過程を要求または公開しない。
- 確認された事実と設計アイデアを同じ確定度で述べない。

## 10. 中核疑似コード

```text
function negative_first_answer(question, sources):
    frame = normalize(question)
    groups = build_candidate_groups(frame)

    for group in groups:
        candidates = activate(group)

        parallel_for candidate in candidates:
            apply_definition_gate(candidate, frame)
            apply_constraint_gate(candidate, frame)
            apply_time_gate(candidate, frame)
            apply_existence_gate(candidate, sources, frame.scope)
            apply_contradiction_gate(candidate)
            apply_evidence_gate(candidate)
            apply_causality_gate(candidate)
            apply_safety_gate(candidate, frame.risk)

        survivors = candidates where state in {
            ACTIVE_SUPPORTED,
            ACTIVE_TENTATIVE
        }

        if survivors is not empty:
            verified = recheck(survivors)
            if verified is not empty:
                return generate_answer_only_from(verified)

        zero_state = classify_zero(candidates, frame.scope)

        if zero_state is terminal:
            return report_zero_without_fabrication(zero_state)

    return {
        status: UNKNOWN_UNRESOLVED,
        answer: "現在確認可能な範囲では、判断に必要な根拠が不足している。"
    }
```

## 11. スキル実行トリガー

ユーザーが次のように依頼した場合、このスキルを活性化する。

- 「事実か確認して」
- 「ないことを先に確認して」
- 「可能な原因をすべて検討して」
- 「ハルシネーションなしで答えて」
- 「文書全体から探して」
- 「条件に合わないものを先に除外して」
- 「分からなければ不明と確定して」
- 「このメッシュ方式で答えて」
- 「Negative-First方式で分析して」

明示的な依頼がなくても、事実誤りのリスクが高い、または候補空間が大きい問題に適用できる。

---

## 12. 完了基準

回答前に内部的に次を確認する。

- 質問の範囲が定義されているか。
- 正確な候補と類似候補を区別したか。
- 該当しない候補を先に除外したか。
- 根拠不足と偽を区別したか。
- 候補がゼロになった場合、次の合理的なグループを確認したか。
- 不在の主張を閉じた範囲に限定したか。
- 生き残った候補だけで答えたか。
- 不確実性がユーザーに見えるか。
- 最新性が必要な事実を検証したか。
- 根拠の空白を想像で埋めていないか。

該当するすべての基準を満たした場合にのみ完了とする。
