# Negative-First Mesh Reasoning

[**한국어**](SKILL.ko.md) | [English](SKILL.en.md) | [日本語](SKILL.ja.md) | [简体中文](SKILL.zh-CN.md)

## 1. 목적

이 스킬은 질문에 대해 가장 그럴듯한 답을 즉시 생성하는 대신 다음 순서로 동작한다.

1. 질문을 검증 가능한 조건과 의미 축으로 분해한다.
2. 가능한 답·해석·원인·문서·대상을 후보 노드로 활성화한다.
3. 모든 후보에 부정 게이트를 적용해 명백히 아닌 후보를 먼저 끈다.
4. 남은 후보가 없으면 다음 관련 그룹으로 탐색 범위를 확장한다.
5. 충분한 근거가 남았을 때만 답을 생성한다.
6. 끝까지 근거가 없으면 내용을 만들어내지 않고 부재 또는 불확실성을 명시한다.

핵심 원칙:

> 찾은 것보다 먼저 아닌 것을 제거하고, 남은 것만 말한다.

---

## 2. 사용 조건

다음 작업에서 우선 사용한다.

- 사실 여부 확인
- 여러 해석이 가능한 질문
- 원인 분석과 장애 진단
- 파일·메일·문서·데이터 검색
- 기술 또는 제품 비교
- 조건에 맞는 대상 탐색
- 법률·의료·금융처럼 잘못된 확정이 위험한 질문
- 최신 정보가 필요한 질문
- AI가 근거 없이 빈칸을 메울 위험이 있는 질문
- “없다”, “모른다”, “가능하지 않다”를 정확히 구분해야 하는 질문

다음 작업에는 과도하게 적용하지 않는다.

- 순수 창작
- 사용자가 제공한 문장의 단순 번역·교정
- 정답이 명백한 단순 계산
- 자유로운 아이디어 발산만 요청한 경우

## 3. 상태 모델

각 후보 노드는 반드시 다음 상태 중 하나를 가진다.

| 상태 | 의미 |
|---|---|
| `ACTIVE_SUPPORTED` | 근거가 있으며 현재 답 후보로 유지 |
| `ACTIVE_TENTATIVE` | 일부 근거는 있으나 추가 검증 필요 |
| `OFF_FALSE` | 질문 조건과 명백히 불일치 |
| `OFF_CONTRADICTED` | 더 강한 근거나 내부 모순으로 제거 |
| `OFF_OUT_OF_SCOPE` | 현재 검증 범위 밖이므로 제거 |
| `OFF_UNSUPPORTED` | 주장할 근거가 없어 제거 |
| `UNKNOWN_UNRESOLVED` | 참·거짓을 판정할 자료가 부족 |
| `NOT_FOUND_IN_SCOPE` | 정의된 폐쇄 범위에서 검색했으나 발견되지 않음 |

`OFF_UNSUPPORTED`를 `OFF_FALSE`로 바꾸지 않는다.  
근거 부족은 거짓의 증명이 아니다.

## 4. 실행 절차

### 단계 A. 질문 정규화

질문을 다음 요소로 분해한다.

- `target`: 무엇을 찾거나 판정하는가
- `claim`: 어떤 주장을 검증하는가
- `constraints`: 반드시 만족해야 하는 조건
- `exclusions`: 제외해야 하는 조건
- `scope`: 데이터·시간·지역·문서·시스템 범위
- `freshness`: 최신 검증이 필요한가
- `risk`: 틀렸을 때의 위험도
- `answer_type`: 사실, 추정, 추천, 설계, 진단 중 무엇인가

사용자가 말하지 않은 범위를 임의로 무한 확장하지 않는다.  
필수 범위가 빠졌지만 질문 없이 진행 가능한 경우 합리적인 기본 범위를 선언하고 진행한다.

### 단계 B. 그물망 축 생성

후보를 한 줄 목록이 아니라 서로 다른 축으로 펼친다.

기본 축:

1. 정확한 일치
2. 동의어·유사 표현
3. 상위·하위 분류
4. 시간 차이
5. 지역·대상 차이
6. 원인과 결과
7. 반례와 모순
8. 출처와 신뢰도
9. 사용자의 숨은 전제
10. 대안 설명

각 축의 후보를 가능한 한 배치 검색·병렬 조회·동시 비교한다.

### 단계 C. 부정 게이트 우선 실행

다음 게이트를 가능한 후보 전체에 먼저 적용한다.

#### 1) 정의 게이트
질문에서 말하는 대상과 후보의 정의가 같은가?

다르면 `OFF_FALSE` 또는 `OFF_OUT_OF_SCOPE`.

#### 2) 조건 게이트
필수 조건 하나라도 명백히 위반하는가?

위반하면 즉시 `OFF_FALSE`.

#### 3) 시간 게이트
요청 시점과 자료 시점이 맞는가?

오래되었거나 시점이 다르면 `OFF_OUT_OF_SCOPE` 또는 `ACTIVE_TENTATIVE`.

#### 4) 존재 게이트
확인 가능한 출처·파일·데이터에 실제로 존재하는가?

폐쇄된 범위에서 없으면 `NOT_FOUND_IN_SCOPE`.  
웹 전체나 현실 전체처럼 열린 범위에서는 단순히 `UNKNOWN_UNRESOLVED`.

#### 5) 모순 게이트
후보가 알려진 사실, 사용자 조건, 다른 강한 근거와 모순되는가?

모순되면 `OFF_CONTRADICTED`.

#### 6) 근거 게이트
답에 포함할 만큼 직접적인 근거가 있는가?

없으면 `OFF_UNSUPPORTED`.

#### 7) 인과 게이트
상관관계를 원인으로 잘못 말하고 있지 않은가?

인과가 검증되지 않으면 원인 후보를 `ACTIVE_TENTATIVE` 이하로 낮춘다.

#### 8) 안전 게이트
잘못 확정했을 때 위험한가?

위험도가 높으면 더 강한 근거 없이는 확정하지 않는다.

### 단계 D. 0 감지

모든 후보가 꺼졌는지 확인한다.

#### 활성 후보가 1개 이상
남은 후보의 근거 강도와 모순 여부를 재검사한 뒤 답을 구성한다.

#### 활성 후보가 0개
즉시 내용을 만들어내지 않는다. 다음을 구분한다.

- 조건 자체가 모순됨 → `CONTRADICTED_QUERY`
- 폐쇄된 검증 범위에서 없음 → `NOT_FOUND_IN_SCOPE`
- 검색 범위가 부족함 → 다음 그룹으로 확장
- 자료 부족으로 판단 불가 → `UNKNOWN_UNRESOLVED`
- 주장이 명백히 틀림 → `FALSE`

### 단계 E. 다음 그룹 확장

첫 그룹에서 답이 없을 때만 다음 순서로 확장한다.

1. 정확한 표현
2. 철자·표기 변형
3. 동의어와 약어
4. 상위·하위 개념
5. 인접 시간대
6. 인접 데이터 소스
7. 반대 방향 검색
8. 원인 대신 결과, 결과 대신 원인
9. 사용자 전제의 대안 해석

확장할 때마다 원래 질문과의 거리를 기록한다.  
질문에서 너무 멀어진 후보는 답처럼 제시하지 않는다.

### 단계 F. 답 생성

답은 남아 있는 `ACTIVE_SUPPORTED`와 필요한 경우 명시적으로 표시된
`ACTIVE_TENTATIVE`만 사용해 생성한다.

기본 순서:

1. 직접 답
2. 판정 상태
3. 적용한 검증 범위
4. 핵심 근거
5. 제거된 주요 오해 또는 후보
6. 남아 있는 불확실성
7. 다음에 확인해야 할 그룹

사용자에게 내부 추론 전체나 숨은 사고과정을 노출하지 않는다.  
대신 검증 가능한 짧은 제거 요약과 근거만 보여준다.

## 5. 부재와 무지 판정 규칙

다음 문장을 엄격히 구분한다.

### “거짓이다”
반례나 직접 증거로 주장이 성립하지 않음이 확인된 경우에만 사용한다.

### “검증 범위 내에서 찾지 못했다”
검색 대상이 명확히 닫힌 범위일 때 사용한다.

예:
- 제공된 문서 3개
- 특정 데이터베이스
- 지정된 날짜 범위
- 사용자의 받은편지함
- 특정 저장소의 현재 브랜치

### “확인할 수 없다”
자료가 없거나 접근할 수 없어 판단할 수 없을 때 사용한다.

### “가능성이 낮다”
직접 부정 증거는 없지만 여러 조건과 맞지 않을 때 사용한다.

### “모른다”
다음 그룹까지 합리적으로 탐색했지만 판정 근거가 남지 않았을 때 사용한다.
단순히 첫 검색에서 안 보였다는 이유로 사용하지 않는다.

## 6. 신뢰도 계산

내부적으로 다음 항목을 0~1로 평가할 수 있다.

- `D`: 직접 근거
- `S`: 출처 신뢰도
- `C`: 조건 충족도
- `R`: 최신성
- `X`: 모순 패널티
- `G`: 질문과 후보 사이 거리 패널티

권장 점수:

`confidence = D × S × C × R × (1 - X) × (1 - G)`

점수는 절대적 확률이라고 주장하지 않는다.

권장 판정:

- `0.85 이상`: 강한 근거
- `0.65~0.84`: 상당히 가능
- `0.40~0.64`: 불확실
- `0.40 미만`: 답 후보에서 제거하거나 추정으로만 표시

고위험 분야는 임계값을 더 높인다.

## 7. 도구 사용 규칙

### 웹
최신 정보, 법률, 가격, 일정, 정치·공직자, 소프트웨어 버전, 현재 회사 정보는
기억으로 답하지 말고 최신 출처를 확인한다.

### 파일
질문이 특정 파일·문서·첨부물에 의존하면 웹보다 파일 원문을 우선 검색한다.

### 이메일·캘린더·저장소
사용자 개인 데이터에 관한 질문은 해당 연결 도구의 실제 범위 안에서만
“없음”을 확정한다.

### 검색 최적화
도구가 허용하면 후보 그룹을 한 번에 배치 조회한다.
순차 호출이 필요해도 논리적으로는 동일 후보 집합에 같은 부정 게이트를 적용한다.

### 인용
사실 판단에 사용한 핵심 근거를 답 가까이에 인용한다.
근거 없는 추정을 사실처럼 말하지 않는다.

## 8. 사용자 응답 형식

### 기본형

```text
[답]
가장 근거가 강한 결론.

[판정]
확인됨 / 상당히 가능 / 불확실 / 검증 범위 내 없음 / 판단 불가 / 거짓

[범위]
무엇을 어디까지 확인했는지.

[먼저 제거된 것]
핵심적으로 아닌 후보 1~3개.

[남은 불확실성]
아직 확인되지 않은 부분.
```

사용자가 간단한 답을 원하면 `[답]`과 필요한 불확실성만 자연스럽게 전달한다.

### JSON 출력이 필요한 경우

```json
{
  "answer": "",
  "status": "CONFIRMED | LIKELY | UNCERTAIN | NOT_FOUND_IN_SCOPE | UNKNOWN | FALSE",
  "scope": [],
  "active_candidates": [],
  "disabled_candidates": [
    {
      "candidate": "",
      "state": "OFF_FALSE | OFF_CONTRADICTED | OFF_OUT_OF_SCOPE | OFF_UNSUPPORTED",
      "reason": ""
    }
  ],
  "evidence": [],
  "confidence": 0.0,
  "next_group": []
}
```

## 9. 금지 행동

- 답을 먼저 만든 뒤 근거를 끼워 맞추지 않는다.
- 첫 검색 실패를 현실 전체의 부재로 확대하지 않는다.
- `근거 없음`을 `거짓`으로 바꾸지 않는다.
- 하나의 유사 후보를 정확한 답으로 둔갑시키지 않는다.
- 오래된 정보를 최신 사실처럼 사용하지 않는다.
- 후보가 모두 꺼졌을 때 언어적으로 그럴듯한 내용을 생성하지 않는다.
- 무한 탐색을 주장하지 않는다.
- 물리적·계산적 한계를 숨기지 않는다.
- 사용자에게 숨은 사고과정 전체를 요구하거나 노출하지 않는다.
- 확인된 사실과 설계 아이디어를 섞어 같은 확정도로 말하지 않는다.

## 10. 핵심 의사코드

```text
function negative_first_answer(question, sources):
    frame = normalize(question)
    groups = build_candidate_groups(frame)

    for group in groups:
        candidates = activate(group)

        parallel_for candidate in candidates:
            apply_definition_gate(candidate, frame)
            apply_constraint_gate(candidate, frame)
            apply_time_gate(candidate, frame)
            apply_existence_gate(candidate, sources, frame.scope)
            apply_contradiction_gate(candidate)
            apply_evidence_gate(candidate)
            apply_causality_gate(candidate)
            apply_safety_gate(candidate, frame.risk)

        survivors = candidates where state in {
            ACTIVE_SUPPORTED,
            ACTIVE_TENTATIVE
        }

        if survivors is not empty:
            verified = recheck(survivors)
            if verified is not empty:
                return generate_answer_only_from(verified)

        zero_state = classify_zero(candidates, frame.scope)

        if zero_state is terminal:
            return report_zero_without_fabrication(zero_state)

    return {
        status: UNKNOWN_UNRESOLVED,
        answer: "현재 확인 가능한 범위에서는 판단할 근거가 부족하다."
    }
```

## 11. 스킬 실행 트리거

사용자가 다음과 같이 요청하면 이 스킬을 활성화한다.

- “사실인지 확인해줘”
- “없다는 것을 먼저 확인해줘”
- “가능한 원인을 모두 검토해줘”
- “환각 없이 답해줘”
- “문서 전체에서 찾아줘”
- “조건에 맞지 않는 것을 먼저 제거해줘”
- “모르면 모른다고 확정해줘”
- “이 그물망 방식으로 답해줘”
- “Negative-First 방식으로 분석해줘”

명시적 요청이 없어도 사실 오류의 위험이 크거나 후보가 많은 문제에서는 적용할 수 있다.

## 12. 완료 기준

답변 전에 내부적으로 다음을 확인한다.

- 질문의 범위가 정의되었는가?
- 정확한 후보와 유사 후보를 구분했는가?
- 아닌 후보를 먼저 제거했는가?
- 근거 부족과 거짓을 구분했는가?
- 후보가 0일 때 다음 그룹을 합리적으로 확인했는가?
- 부재 주장이 닫힌 범위에 한정되어 있는가?
- 남은 후보만으로 답했는가?
